## 内部思考要求

在正式输出前，你必须先完成以下业务建模判断，但**不要把这段思考过程直接写出来**，而是将结果体现在最终方案中：

1. 先判断该场景属于哪类业务，可多类并存：
   - 基础资料管理 / 审批流转 / 库存仓储 / 采购销售 / 借还租赁 / 生产加工 / 客户服务工单 / 项目任务协同 / 财务结算 / 运营分析

2. 识别该场景是否存在关键业务规则、控制机制或异常处理要求：
   - 状态流转 / 审批控制 / 时效约束 / 配额限制 / 批次管理 / 序列号管理 / 先进先出 FIFO / 临期先出 FEFO / 效期管理 / 冻结预留 / 部分执行 / 退回驳回撤销作废 / 退货归还回冲 / 对账结算 / 追溯留痕审计 / 盘盈盘亏差异 / 丢失损坏逾期补录等异常

3. 若存在上述规则，必须落实到系统结构中（不能只写在描述里），体现为：必要的工作表 / 业务流程 / 自定义动作 / 自动化工作流 / 统计分析入口

4. 必须覆盖完整业务闭环：发起 → 处理 → 审批执行 → 结果登记 → 异常处理 → 统计分析

5. 工作表设计不能只停留在"主业务对象"层面，还应判断是否需要补齐：
   - 基础资料对象 / 业务单据对象 / 过程明细对象 / 状态流转对象 / 库存台账流水对象 / 异常处理对象 / 统计分析支撑对象

6. 不得为控制表数量省略关键过程对象、流水对象、异常记录对象或分析支撑对象。若某业务规则需要额外对象才能成立，优先保证业务闭环。

## 设计原则

- 优先体现"为什么这样设计"，而不只是"包含什么"
- 目标是生成覆盖**完整业务链**的真实可用方案，不因用户描述简短就缩减结构，应补齐基础资料、业务过程、审批流转、异常处理、统计分析等关键环节
- 对需要触发工作流的自定义动作，必须同步规划对应的工作流
- 根据业务场景，提供丰富且合理的自定义动作，尽可能简化用户高频操作
- 自定义动作原则：按钮出现在「操作发起方」的工作表上，而非即将被创建/被操作的目标表上。例：「发起借阅」按钮在「图书」表上；「归还」按钮在「借阅记录」表上
- **审批类操作不要拆成自定义动作**。「审批通过」「审批驳回」等操作应由工作流中的审批块（approve 节点）处理，通过/驳回后的不同逻辑用分支节点区分。正确做法：记录创建后自动触发一条包含审批环节的工作流，而非为通过和驳回各建一个按钮
- 若场景存在行业关键规则（如 FIFO、批次追溯、冻结库存、驳回重提、逾期归还、部分发货等），必须体现在方案结构中，而不只是一句描述
- **本阶段不展开**：详细字段、视图、图表配置、工作流节点步骤等细节，这些在后续搭建阶段补全

### 典型业务场景补齐要求

当场景涉及以下类型时，应自动补足常见控制对象与机制：

- **库存 / 仓储类**：仓库、批次/效期、库存流水、盘点、调拨、冻结/预留、FIFO/FEFO
- **采购 / 销售类**：订单、收发货、退货、对账、结算、供应商/客户协同
- **借还 / 租赁类**：借出、归还、逾期、续借/续租、损坏/遗失处理、在借状态控制
- **审批类**：申请、审批、驳回、撤回、重新提交、审批留痕
- **服务工单类**：受理、分派、处理、回访、关闭、超时升级
- **项目 / 任务类**：计划、执行、延期、验收、变更、风险跟踪

4～6 条，概括整体组织逻辑，必须满足的关键业务规则、控制机制或异常处理原则，并简要说明这些规则会如何体现在工作表、流程、自定义动作或自动化工作流中。

# 3. 角色（Role）

每个角色包含：名称 + 1～2 句职责说明 + 主要参与方式。

# 4. 业务流程（Process & Information）

拆解核心流程，每个流程包含：流程名称 + 1～2 句说明。体现前后衔接和业务闭环。

# 5. 应用内容（Content）

## 工作表

每张表包含：**名称** + 1～2 句说明；当需要自定义动作时，列出动作名称。

工作表之后附 Mermaid ER 图，展示核心表的关联关系：

```mermaid

    "工作表A" ||--o{ "工作表B" : "关联关系描述"
```

## 自定义页面 (custompage)

每个页面包含：**名称** + 页面类型（`dashboard` 或 `workspace`）+ 1～2 句说明；标注适用角色。

## 自动化工作流

每条工作流包含：**名称** + 触发方式 + 触发场景与业务目的。

## AI 助手

仅在智能问答、数据查询或辅助分析需求时规划，不强制。

## 导航分组

每个分组包含：**分组名称** + 所含工作表/仪表盘/AI助手名称列表。


### 格式规范

- 各类对象均使用列表格式，名称**加粗**
- 每段说明控制在 1～2 句话
- 对象名称本身不加反引号；在说明文字中引用时加反引号
- 直接输出内容，不加客套话或解释性前言

# HAP 应用设计指南

> 本文档覆盖 HAP 应用的**结构设计**（建哪些对象、如何组织）与**内容设计**（字段、视图、图表、工作流、自定义动作的具体规范）。所有设计决策必须严格遵循本指南。

---

## 零、应用命名规范

应用名称代表整个系统的业务边界与专业度，应保持极致简洁、业务场景聚焦、富现代感，并遵循以下原则：

1. **剔除冗余后缀**：**严禁**使用“系统”、“平台”、“软件”、“工具”、“应用”、“表”、“系统后台”等传统而笨重的后缀。
   - ❌ 传统冗长：`SaaS 客户订阅与服务生命周期管理系统`、`重型设备维保管理平台`
   - ✅ 现代极简：`SaaS 客户订阅与服务`、`设备维保与健康监测`、`图书借阅大厅`
2. **场景聚焦与字数限制**：应用名称建议控制在 **2~8 个字** 之间，极其精准地突显核心业务实体与动作概念本身。

---

## 应用设计顺序

设计一个 HAP 应用时，应按以下顺序推进。每一步都依赖上一步的结论：

| 步骤 | 模块 | 核心问题 |
|------|------|----------|
| 1 | **角色权限** | 谁使用这个应用？各角色职责和权限边界是什么？ |
| 2 | **工作表** | 需要管理哪些业务对象？数据模型如何支撑各角色的工作？ |
| 3 | **自动化工作流** | 哪些流程节点可以自动化？哪些需要人工触发？ |
| 4 | **自定义页面** | 管理者需要看到什么数据看板？一线操作者需要什么工作台入口？ |
| 5 | **导航分组** | 如何将上述所有内容组织成用户友好的入口层级？ |
| 6 | **外部门户** | 是否有外部用户需要受限访问？ |
| 7 | **AI 助手** | 是否有明确的智能查询或数据分析需求？ |

> **全局原则——以角色驱动设计**：角色是贯穿整个应用的设计主线。每个视图、自定义页面、工作流、自定义动作都应先明确“服务于哪个角色”，再决定内容。具体表现为：
> - **视图**：同一张工作表，不同角色看到不同的视图（数据范围、筛选条件、展示字段各异）
> - **自定义页面**：不同角色拥有各自的仪表盘或工作台（管理者看经营分析，操作者看待办任务）
> - **自定义动作**：按钮应只对有权操作的角色可见
> - **工作流通知**：通知对象应精确到角色，避免全员广播

## 一、角色权限（Role）

确定谁使用这个应用，以及他们能看到什么、操作什么。角色基于 RBAC 原则设计，用于定义数据范围、操作权限、审批责任和业务入口。

> **先想清楚「有哪些业务责任」，再决定「需要哪些数据权限」。**
> 角色不是组织架构，也不是岗位名单，而是一组可复用的业务权限能力。HAP 支持一个用户拥有多个角色，因此应优先拆分清晰的业务角色，再通过多角色组合覆盖复合岗位。

---

### 系统预设角色

系统已包含以下预设角色，**不需要重复创建**：

| 预设角色  | 权限范围             |
| ----- | ---------------- |
| 应用管理员 | 所有权限（数据 + 应用配置）  |
| 运营者   | 所有数据权限 + 角色/用户管理 |
| 搭建者   | 应用编辑权限（结构搭建）     |

设计时只需创建**业务角色**，无需创建「系统管理员」「超级管理员」「平台管理员」等系统级角色。

---

### 角色设计原则

#### 1. 按"业务职责"命名，而非"组织层级"

| 推荐         | 不推荐         | 原因            |
| ---------- | ----------- | ------------- |
| 借阅人员、图书管理员 | 普通员工、部门成员   | 层级描述，权限边界不清晰  |
| 采购专员、财务审核  | 采购部、财务部     | 部门不等于角色       |
| 销售代表、风控专员  | 业务人员、后台人员   | 职责过泛，无法指导权限设计 |
| 维修厂、供应商 、读者   | 外部用户、外部人员   | 应按外部协作职责命名    |
| ——         | 系统管理员、超级管理员 | 已由系统预设角色覆盖    |

#### 2. 按"权限差异"决定是否拆分

当角色在以下任一方面存在差异时，就应当拆分为不同角色：

* 能看到的数据范围不同
* 能操作的工作表不同
* 能执行的动作不同
* 参与的审批节点不同
* 使用的视图 / 自定义页面入口不同
* 内部角色与外部角色

#### 3. 利用"多角色组合"，避免创建过度复合角色

HAP 支持一个用户拥有多个角色，不要为了某个人的复合职责创建混合角色。

| 不推荐       | 推荐           |
| --------- | ------------ |
| 销售主管兼客户经理 | 销售主管 + 客户经理  |
| 采购审批兼财务审核 | 采购审批人 + 财务审核 |
| 咨询员兼面谈员   | 咨询员 + 面谈评估员  |

#### 4. 识别外部角色

当业务流程中可能需要**非本组织成员**参与协作时，应设计专门的外部角色。外部角色通过 HAP 外部门户访问应用，数据天然隔离。

识别信号：
- 合作方、供应商、客户、加盟商、外部读者等需要在系统中填报、查看或确认数据
- 业务流程中存在"跨组织边界"的协作环节（如维修厂接单、供应商报价、客户查看进度）
- 需要限制数据可见范围到"仅与自己相关的记录"
- 当同一角色既有内部协作关系，又有外部协作关系时（如维修厂既有内部管理人员，又有外部维修人员）应设计两个角色

外部角色默认只能看到与自己关联的记录，通常不参与内部审批流程，但可以作为流程的发起方或接收方。

---

### 权限层级参考

| 层级   | 数据范围                   | 适合角色              |
| ---- | ---------------------- | ----------------- |
| 只读   | 查看指定范围记录，不能修改          | 管理层、观察者、汇报对象      |
| 操作员  | 新增、编辑自己负责的记录           | 一线录入人员、业务经办人      |
| 协同处理 | 查看并处理与自己相关的记录          | 维修厂、供应商、客服等外部协作方  |
| 主管   | 查看和编辑团队或本机构范围内记录       | 团队负责人、设施管理者、部门负责人 |
| 审核人  | 查看待审核记录，并执行审批、驳回、确认等动作 | 财务审核、风控专员、法务审核    |

---

### 完整性检查

生成业务角色后，按以下清单逐项检查是否遗漏：

| 角色类型   | 说明                    | 示例                 |
| ------ | --------------------- | ------------------ |
| 一线操作角色 | 负责日常录入、跟进、处理业务数据      | 销售代表、咨询员、仓管人员、维保专员 |
| 业务管理角色 | 负责团队监督、进度跟踪、异常处理      | 销售主管、设施管理者、车队管理员   |
| 审批审核角色 | 负责关键节点审批、复核、确认        | 财务审核、风控专员、采购审批人    |
| 专业支持角色 | 负责某类专业业务处理            | 法务专员、评估员、财务专员      |
| 外部协作角色 | 通过门户或受限入口参与业务（外部角色）   | 维修厂、供应商、客户、合作机构    |
| 只读分析角色 | 查看统计、报表、驾驶舱，不直接操作数据   | 管理层、法人管理者、观察者      |

不是每个应用都必须包含所有类型，但需要根据业务流程判断是否需要。注意"管理层"通常应优先作为只读分析角色，除非其确实参与审批或业务处理。

## 二、工作表（Worksheet）

数据是应用的基础。工作表设计决定了整个应用的信息结构。

### 两类工作表

| 类型 | 特征 | 典型例子 |
|------|------|----------|
| **基础资料类**（Master Data）| 主数据，相对稳定，被其他表引用 | 客户、图书、设备、员工、商品 |
| **业务过程类**（Transactional Data）| 流程记录，随业务持续产生 | 订单、申请、借阅记录、巡检记录 |

### 命名准则

- **基础资料类**：直接用业务对象名（名词单数）。✅ `图书`、`客户` &emsp; ❌ `图书列表`、`客户信息表`
- **业务过程类**：对象 + 动作名词。✅ `借阅记录`、`采购申请`、`维修工单`

### 建表判断

**建表前问自己**：这个对象需要独立的生命周期管理吗（需增删改查）？

- **是** → 独立建表，通过关联字段引用
- **否**（只是分类标签）→ 用单选 / 多选字段，不建表

### 核心原则

- **单一数据源**：同一业务对象只建一张表，用字段区分状态，用视图区分角色视角，用权限控制访问范围
- **状态机思维**：业务对象的不同阶段是同一张表里的状态字段。"待审批"和"已审批"是状态值，不是两张表

### 误用边界

- 不要按角色拆表——权限差异通过角色配置解决
- 不要把流程阶段拆成多张独立表
- 分类不需要单独管理时，用单选字段，不建表

## 三、字段（Field）

### 字段完整性

设计一张工作表字段时，按以下 7 个维度依次补齐。**每个维度至少覆盖 1 个字段**，不得遗漏任何维度：

| 维度 | 说明 | 典型字段示例 |
|---|---|---|
| ① 主标识 | 名称、编号、标题——让用户一眼识别这条记录 | 书名、订单编号、工单标题 |
| ② 核心业务属性 | 该业务对象的**核心描述信息**，体现行业特征 | ISBN、出版社、页数、作者（图书）；规格、品牌、产地（商品）；联系人、行业（客户） |
| ③ 状态与分类 | 状态流转、类型分类、优先级 | 借阅状态、图书分类、紧急程度、审批状态 |
| ④ 时间维度 | 创建时间、截止日期、完成时间、有效期等 | 借出日期、应还日期、实际归还日期、入库日期、创建时间 |
| ⑤ 责任与协作 | 负责人、创建人、审批人、操作人 | 经办人、管理员、借阅人、审批人 |
| ⑥ 数量与金额 | 数量、金额、统计值——支撑工作流的加减运算 | 库存数量、单价、总金额、累计次数 |
| ⑦ 说明与凭证 | 备注、附件、图片、富文本说明 | 图书封面、备注、合同附件、商品图片、图书简介 |

> ⚠️ **维度 ② 是最容易被忽略的维度**。很多设计只有名称+状态+时间，缺少真正体现业务对象特征的属性字段。例如「图书」表只有书名和作者，缺少 ISBN、出版社、出版日期、页数、语言、馆藏位置等关键业务属性。

### 业务场景字段补全

根据工作表所属的业务场景，主动补齐以下常见但容易遗漏的字段：

**主数据 / 资料类**（客户、图书、设备、商品、员工）：
- 图片/封面（Attachment）、简介/描述（RichText）
- 所在位置/地址（Location 或 Text）
- 启用状态（SingleSelect 或 Checkbox）、创建时间（DateTime）
- 行业特征属性（如图书的 ISBN/出版社/页数；商品的规格/品牌/产地；设备的型号/序列号/购入日期）

**单据 / 记录类**（订单、借阅记录、工单、入库单）：
- 单据编号（AutoNumber）、操作人（Collaborator）、操作时间（DateTime）
- 处理状态（SingleSelect，**必须包含完整状态链**，覆盖从创建到结束的所有状态）
- 来源/渠道（SingleSelect）

**涉及库存 / 数量的表**：
- 总量、已用量/在借量、可用量/剩余量（Number）
- 确保能支撑工作流中的加减运算（如借出时 `可借库存 -1`，归还时 `可借库存 +1`）

**涉及审批 / 流转的表**：
- 审批状态（SingleSelect）、审批人（Collaborator）、审批时间（DateTime）、审批意见（Text）

### 选型规则

* 自由输入短文本 → `Text`
* 带格式长文本 → `RichText`
* 普通数值 → `Number`
* 金额 → `Currency`
* 固定单值枚举 → `SingleSelect`
* 固定多值枚举 → `MultipleSelect`
* 选项较多需搜索 → `Dropdown`
* 只要日期、年、年月 → `Date`
* 要具体时点 → `DateTime`
* 只要时间（不含日期） → `Time`
* 单个是/否 → `Checkbox`（不要用 SingleSelect + 是/否）
* 1～10 级评分 → `Rating`
* 手机号 → `PhoneNumber`
* 座机/电话 → `LandlinePhone`
* 邮箱 → `Email`
* 定位坐标 → `Location`
* 省市区行政区域 → `Region`
* 负责人/审批人 → `Collaborator`
* 部门归属 → `Department`
* 组织角色 → `OrgRole`
* 跨表引用/联动/统计 → `Relation`
* 本表父子层级（上级任务/父分类）→ `Relation`（dataSource=selfRelation）
* 数值计算（仅支持+-*/（）数学运算） → `Formula`
* 日期计算/差值/偏移 → `DateFormula`
* 单据流水号 → `AutoNumber`
* 材料/图片/文件 → `Attachment`

### 关联字段规则

仅在以下情况建立 `Relation`：

* 需要引用另一张表记录
* 需要与另一张表联动
* 需要基于另一张表筛选、统计、回写
* 需要表达本表父子层级

### 误用边界

* 状态、类型、优先级不要用 `Text`
* 负责人不要用姓名文本，优先用 `Collaborator`
* 不要把所有字段都做成 `Text`
* 不要仅因"查看方便"建立关联
* 不要过度设置必填字段
* 不要只生成总览里提到的几个字段就停止

## 四、视图（View）

### 默认设计规则

每张工作表至少应有 1 个Table 类型的 `全部` 视图：用于完整浏览、筛选、排序、分组、批量处理

根据工作表的业务场景、字段结构与角色查看需要，再增加第 2～5 个视图。

### 字段、业务驱动的视图

当工作表的业务和字段结构与某类视图高度契合时，**必须**主动添加该类型视图

| 字段条件 | 必须添加的视图 | 说明 |
|---|---|---|
| 存在 selfRelation 或指向本表的 Relation | `Hierarchy` | 天然的父子树结构，用 Table 无法直观展示层级 |
| 存在状态/阶段类 SingleSelect（选项 ≥ 3） | `Kanban` | 状态流转是看板的核心场景 |
| 存在 Location 定位字段 | `Map` | 地理分布只有地图视图能直观呈现 |
| 附件字段是表的核心内容（如产品图、案例封面） | `Gallery` | 图片化浏览是 Gallery 的不可替代优势 |
| 表的业务语义是任务/排期，且有计划开始+结束日期 | `Gantt` | 项目排期、任务计划等需要时间轴视角 |
| 表的业务语义是排班/预约，且有时间段+人员/资源字段 | `Resource` | 资源占用与时间冲突只有资源视图能直观呈现 |
| 表是参数配置/系统设置，通常只有 1 条记录 | `Detail`（mode=first） | 单记录场景无需列表，直接展示详情 |

### 按角色设计视图

不同角色对同一张工作表的关注点不同，应通过视图区分角色视角，而非按角色拆表：

* 一线操作者 → 「我的待处理」「我负责的」等筛选型视图，只看自己相关的数据
* 业务管理者 → 「全部记录」「团队待办」等全局视图，用于监控和分配
* 审批角色 → 「待我审批」视图，按审批状态筛选
* 只读分析角色 → 按需分配已有视图的查看权限即可，通常不需要专属视图

### 选型规则

* 批量浏览、筛选、编辑、导出 → `Table`
* 当前人只看自己数据 → 增加筛选型 `Table` 视图，如 `我的待处理`
* 状态泳道、拖拽推进 → `Kanban`
* 看某天有什么 → `Calendar`
* 看任务起止与排期 → `Gantt`
* 看图片内容（产品、案例、简历）→ `Gallery`
* 看单条完整信息 → `Detail`
* 看父子结构 → `Hierarchy`
* 看地理分布 → `Map`
* 看资源/人员占用与排班 → `Resource`

### 误用边界

* 有日期字段，不代表一定建 `Calendar`
* 有开始/结束日期，不代表一定建 `Gantt`
* 有附件字段，不代表一定建 `Gallery`
* 有关联字段，不代表一定建 `Hierarchy`
* 不要因为"要分组"就一律用 `Kanban`，很多场景更适合 `Table + 分组`
* 每张工作表视图建议不超过 5 个

## 五、自定义动作（Custom Action）

工作表记录上可以配备操作按钮（自定义动作），这是串联业务流程的核心人工入口。

### 三种类型

1. **填写当前记录**（`update`）：将一组字段归类填写，方便用户集中操作。
   > 适用：需要用户**主动输入内容**才能推进流程的场景
   > 例：点击「更新图书信息」按钮，弹出表单让用户填写

2. **新建关联记录**（`create`）：用户点击按钮新建一条关联记录
   > 适用：用户在当前记录上**发起一件需要填写信息的新事务**
   > 例：在「设备」记录上点击「报修」按钮，填写「报修申请」

3. **直接触发工作流**（`trigger`）：不弹出表单，点击后直接触发工作流自动处理
   > 适用：业务推进**不需要用户输入**，系统自动完成
   > 例：借阅记录上的「归还」按钮——状态自动改为已归还、归还日期自动写入今天、库存自动加一

### 挂载规则

动作必须挂在**动作发起的数据源头表**上，不要挂在目标表上。

* 发起借阅 → 挂在图书表，不挂借阅记录表
* 发起工单 → 挂在客户表，不挂工单表
* 登记归还 → 挂在借阅记录表，不挂图书表

> 💡 即使不触发工作流也需要为高频操作（填写记录、新建关联）尽可能提供自定义动作，提高操作便利性。

### 误用边界

* 严禁在没有直接物理关联（Relation）字段的两张表之间，设计并生成类型为 "新建关联记录(create)" 的自定义动作。在规划中定义 `createRelatedRecord` 类型的动作时，**必须同时提供用于物理关联的具体 Relation 字段名称**（该字段必须显式声明在源工作表的 `fields` 列表中）。如果两张表之间在规划中没有直接的 Relation 字段，则禁止将其规划为 `createRelatedRecord`，而应当：
  - (1) 在源工作表字段列表中显式追加并声明 Relation 关联字段，并在动作中引用它；
  - (2) 或者将自定义动作的类型设计为“直接触发工作流(trigger)”，利用后台工作流自动创建对应记录。
* 不要把"新建关联记录"类动作挂在目标表上
* 不要让需要人工输入的场景用 `trigger`
* 不要把本可自动完成的动作做成 `update/create`
* 如果字段值是"今天""当前用户""固定状态"等系统可自动确定值，通常不应要求用户手填
* 不需要人工点击的场景 → 改用自动化工作流，不做自定义动作

---

## 六、自动化工作流（Workflow）

工作流是业务流程的自动化执行，减少重复人工干预。

> **核心原则**：先理清业务流程，再考虑自动化。自动化只是加速，不能替代对流程本身的设计。

### 触发方式选型

| 触发方式 | 适用场景 | 典型应用 |
|----------|----------|----------|
| 工作表事件（`RecordEvent`） | 记录新增 / 编辑 / 删除后立即触发 | 通知推送、状态同步、数据联动 |
| 日期字段（`DateField`） | 基于记录中某日期字段相对触发 | 到期提醒、定期跟进、续费预警 |
| 定时（`Scheduled`） | 固定周期执行 | 逾期检查、周期汇总、盘点提醒 |
| 自定义动作（`CustomAction`） | 用户主动点击按钮后触发 | 审批提交、确认归还、发起退款 |

### 动作节点选型

* 生成新单据/流水 → 新增记录 (`add_record`)
* 推进状态/自动赋值/联动回写 → 更新记录 (`update_record`)
* 删除特定记录 → 删除记录 (`delete_record`)（⚠️ 仅在必要时使用，优先使用更新状态为已作废代替）
* 获取其他表数据 → 查询单条 (`get_single`) / 多条记录 (`get_multiple`)
* 对查询结果统计 → 汇总 (`rollup`)
* 动态计算数值或日期 → 计算 (`compute`)（包含数值公式、日期差值 `dateDiff`、日期偏移 `dateOffset`）
* 触达相关人员 → 站内通知 (`send_internal_notice`) / 抄送 (`cc`) / 发送邮件 (`send_email`)
* 流程等待或延迟 → 延时 (`delay`)（支持等待固定时长或延迟到指定时间）
* 引入人工决策与流转 → 审批业务块 (`approval_block`)（从发起审批开始，内部可包含审批 `approve`、填写 `fill_in`、抄送节点）
* 针对多条记录循环执行 → 子流程 (`sub_process`)（常与查询多条记录 `get_multiple` 搭配使用，遍历记录并逐个处理）
* 按条件分流 → 条件分支 (`branch`)
* 图形节点不足时补逻辑 → 代码块 (`code`)（支持 JS / Python 脚本）


### 误用边界

* 不要用 `Scheduled` 做实时通知（应使用工作表事件触发）
* 不要把高频变化规则写死在节点里，优先用字段值 + 条件分支
* 不要给所有流程都套审批
* 不要只发通知不写回数据
* 删除记录要谨慎，很多场景更适合"更新状态为已作废"
* 含有不确定人工判断的步骤不要自动化，用自定义动作让人来决策

---

## 七、统计图表（Chart）

### 默认设计规则

一个 dashboard 优先覆盖以下 3 类问题：

1. 核心指标总览
2. 时间趋势变化
3. 结构占比或排行对比

只有业务明确需要时，再补充漏斗、透视、雷达、散点等复杂图表。

### 选型规则

* 看关键数字 → `NumberChart`
* 看完成率 / 达成度 → `GaugeChart` / `ProgressChart`
* 看类别对比 → `ColumnChart` / `BarChart`
* 看名称较长的类别对比 → `BarChart`
* 看时间趋势 → `LineChart`
* 看双指标趋势 → `DualAxisChart`
* 看占比构成 → `PieChart`
* 看阶段转化 → `FunnelChart`
* 看排行榜 / TopN → `RankingChart`
* 看双侧群体对比 → `SymmetricBarChart`
* 看多指标综合比较 → `RadarChart`
* 看两个数值变量关系 → `ScatterChart`
* 看多维交叉分析 → `PivotTable`
* 看地区分布（国内） → `RegionMap`
* 看地区分布（全球） → `WorldMap`
* 看文本热点 / 关键词分布 → `WordCloud`

### 图表设计规则

每个图表都必须先明确：

* 回答什么业务问题
* 指标是什么
* 按什么维度看
* 为什么值得进 dashboard 首页

### 误用边界

* 不要多个图表回答同一个问题
* 不要堆过多 `NumberCard` 或 `PieChart`
* `PieChart` 类别过多时改用 `BarChart`
* 没有明确阶段顺序不要用 `FunnelChart`
* 没有连续时间维度不要用 `LineChart`
* 每个 dashboard 建议 6~10 个图表

## 八、自定义页面（CustomPage）

> **页面分类**：自定义页面主要承担两种核心角色：
> 1. `dashboard`：以数据统计、趋势分析为主，聚合工作表数据、展示统计图表、支撑管理决策。
> 2. `workspace`：以快捷操作、任务处理为主，作为应用的门户入口、待办中心或核心业务流的操作台。

### 设计准则

**先明确页面目的**：每个自定义页面应回答一个具体的问题——服务于谁？帮助做什么？

| 页面类型 | 服务对象 | 内容侧重 |
|----------|----------|----------|
| dashboard | 需要看数据做判断的角色（管理者、分析者、一线操作者均可） | 趋势对比、多维交叉分析、聚合统计 |
| workspace | 需要高频操作的角色（一线操作者、全员） | 当前状态、待办处理、高频操作入口 |

**按角色规划页面**：不同角色的关注点和操作方式不同，应为其设计专属或共享的自定义页面：

* 管理者/决策者 → 经营分析仪表盘（核心），辅以管理工作台提供审批、督办等操作入口
* 一线操作者 → 操作工作台（核心），辅以业务仪表盘查看个人绩效、待办统计等数据
* 按角色拆分 → 当不同角色关注的指标或操作有差异时，应分别创建独立的仪表盘或工作台，而非将所有内容合并到一个页面。只有当多个角色的关注内容完全一致时，才共享同一页面

### 页面组件生态 (Component Ecosystem)

自定义页面支持多种组件组合，以满足不同类型页面的需求：

1. **图表类 (Chart)**：如 `NumberChart` (数值)、`BarChart` (柱状图)、`PieChart` (饼图)、`LineChart` (折线图) 等。主要用于聚合展示和分析（`dashboard` 核心）。
2. **操作与数据类**：
   - **快捷按钮 (Button)**：快捷入口，配置创建记录/打开视图/打开页面/打开链接。
   - **内嵌视图 (View)**：将工作表的特定视图（如“待我审批的报销”、“进行中的项目”）直接嵌入页面。
   - 主要用于构建工作台和任务中心（`workspace` 核心）。
3. **展示与布局类**：
   - **富文本 / 网页内嵌 (Text / HTML)**：用于展示操作规范说明、规章制度或外部系统链接。
   - **轮播图 (Carousel)**：展示重点图文资讯、公告。
   - **标题区 (Section)**：用于页面的视觉分区。

### 误用边界

- 不要混淆组件类型定位（如在 workspace 堆砌大量折线图，或在 dashboard 放置过多操作按钮）
- 不同页面应有明确的分工，不重复展示相同数据或入口

---

## 九、导航分组（Navigation Group）

导航结构决定用户进入应用后的第一印象和操作路径。

> **核心原则**：以用户的业务任务和使用场景为主线组织导航分组，兼顾核心业务对象，确保入口清晰、层级稳定、路径自然。

### 推荐结构（从上到下）

| 分组位置 | 推荐内容 | 说明 |
|----------|----------|------|
| 1. 总览 / 首页 | 自定义页面（Dashboard / Workspace）、AI 助手 | 让用户一眼掌握全局状态和核心待办 |
| 2. 核心业务 | 高频操作工作表 | 借阅管理、订单处理、工单跟进 |
| 3. 基础资料 | 主数据工作表 | 低频使用，放靠后 |
| 4. 配置与管理 | 分类维护表、基础数据表 | 极低频，由管理员维护 |

### 分组命名准则

- 优先使用简洁的业务场景词命名；如：借阅 优于 借阅管理，设备 优于 设备运维
- 组名 2～4 字，应简洁、明确
- 同层级分组名称应保持一致的抽象维度

### 结构约束

- 分组通常控制在 3～5 个，避免首层分组过多
- 每组通常承载 2～5 个项目
- 禁止单项分组，只有 1 项时并入相邻分组
- 自定义页面优先归入"总览"或对应业务分组

## 十、AI 助手（AI Assistant）

基于应用内数据提供智能问答、数据查询和分析建议。

### 适用场景

| 适合 | 不适合 |
|------|--------|
| 数据查询："本月库存还剩多少？" | 替代工作流执行自动化操作 |
| 状态查询："我的申请审批到哪步了？" | 代替人工完成界面操作 |
| 辅助分析："哪类图书借阅量最高？" | — |

> 仅在有明确查询或分析需求时规划，不强制每个应用配置。

## 跨模块闭环设计

应用设计中，视图、字段、工作流、自定义动作之间存在强依赖关系。任何一环缺失都会导致功能无效。设计时必须确保以下闭环完整。

### 1. 工作流数据闭环

每条工作流优先形成以下闭环：

1. 更新主表状态
2. 更新关联表统计或状态
3. 新增流水/日志
4. 发送通知

若只有通知、没有数据写回或留痕，通常闭环不足。

### 2. 视图 ↔ 字段闭环

视图的筛选条件必须有对应的字段和选项值支撑。

* ✅ 正确：规划了“超期事项(Table)”视图，状态字段包含“已超期”选项：`"事项状态(SingleSelect:待分解/办理中/已完成/已超期/已归档)"`
* ❌ 错误：规划了“超期事项”视图，但状态字段只有 `"事项状态(SingleSelect:待分解/办理中/已完成/已归档)"`——缺少“已超期”选项，视图无法筛选

### 3. 字段 ↔ 工作流闭环（时效性状态）

当状态字段中存在系统自动判定的选项（如“已超期”“已逾期”“已过期”），该选项无法由用户手动设置，必须有对应的工作流自动标记。

* ✅ 正确：状态字段有“已超期”选项，同时规划了一条 `date_field` 触发的工作流：“完成时限到达时，若事项状态不是已完成，则自动更新为已超期并通知责任人”
* ❌ 错误：状态字段有“已超期”选项，但没有任何工作流来自动标记——该选项永远不会被设置，视图始终为空

典型实现模式：
* `date_field` 触发 → 完成时限到达时，检查是否已完成，若未完成则更新状态为“已超期”
* `schedule` 定时触发 → 每日检查所有未完成记录，将超过完成时限的标记为“已超期”

### 4. 自定义动作 ↔ 字段 ↔ 工作流闭环

自定义动作的前置条件和执行结果都必须有字段支撑，状态变更必须有工作流执行。

* ✅ 正确：“归档”按钮的启用条件为“事项状态=已完成”，点击后触发工作流将状态更新为“已归档”
* ❌ 错误：“归档”按钮没有设置启用条件，任何状态下都可以点击归档，导致未完成的事项也可被归档

### 5. 自定义页面 ↔ 视图 / 字段闭环

工作台中内嵌的视图组件和仪表盘中的图表都依赖工作表字段的完整性。

* ✅ 正确：仪表盘中“超期事项数(NumberChart)”指向事项表，筛选条件为“状态=已超期”，字段和工作流均已配套
* ❌ 错误：仪表盘有“超期事项数”图表，但状态字段没有“已超期”选项，也没有工作流标记超期——图表始终显示 0

---

## 产出质量要求

AI 在设计时应满足：

* 字段：按 7 个维度补齐业务骨架，不滥用 `Text`
* 视图：先有默认 `Table`，再按任务补高价值视图
* 图表：每个图表都回答一个具体业务问题
* 工作流：每条流程尽量形成数据闭环
* 自定义动作：优先判断是否真的需要人工点击与输入
